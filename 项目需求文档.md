## ⚠️ 核心约束：全面兼容 Windows 与 macOS

> **本项目必须在 Windows 和 macOS 上均可正常运行。任何功能设计、代码实现、路径处理都必须同时考虑两个平台。**

| 约束项 | 要求 |
|--------|------|
| **路径处理** | 统一使用 `pathlib.Path`，禁止硬编码 `/` 或 `\` |
| **文件名命名** | 统一小写 + 连字符（slug），禁止使用 Windows 保留字符 `<>:"/\|?*` |
| **目录深度** | ≤ 3 层，确保不超过 Windows MAX_PATH（260 字符）限制 |
| **大小写** | Windows 不区分大小写，文件名统一小写避免冲突 |
| **换行符** | 读写文本文件时使用 `utf-8` + `newline=''`（Python 自动处理 `\r\n`） |
| **Shell 命令** | 不在代码中直接拼接 shell 命令；Hermes 在 Windows 上捆绑 MinGit |
| **环境变量** | 使用 `TRADE_HOME`（macOS: `~/.trade/`，Windows: `%LOCALAPPDATA%\trade\`） |
| **安装方式** | macOS 用 `curl ... | bash`（同 Hermes），Windows 用 PowerShell 脚本 |

**Hermes 上游的 Windows 支持状态**（参照 2026-05 Hermes README）：
- Windows native 为 **Early Beta**，已有 PowerShell 一键安装脚本
- 安装路径：`%LOCALAPPDATA%\hermes\`
- 捆绑 MinGit（便携 Git Bash）用于 shell 命令执行
- WSL2 路径与 Linux 相同（`~/.hermes/`），是"战斗测试最充分"的 Windows 方案

**本项目对 Windows 的定位**：
- 目标：与 macOS 同等的完整功能支持
- 最低要求：Windows 10 及以上
- 终端：PowerShell 5.1+ 或 Hermes 捆绑的 MinGit Bash
- 文件系统：NTFS（默认）或 ReFS

## 一、项目定位

**外贸公司的 AI 销售助手**。不是简单的文档问答工具，而是覆盖「引流 → 转化」全链路的智能销售系统。

传统外贸公司面临两个核心问题：
1. **引流（找客户）**：怎么找到更多潜在客户？
2. **转化（拿订单）**：怎么把潜在客户变成真实订单？

Foreign Trade Assistant 通过训练 Hermes Agent 学习顶级销售员的思维模式和业务技能，帮助外贸业务员在引流和转化两个环节提效。

**核心架构**：Agent Loop — LLM 自主决定读什么文件、搜什么信息、生成什么内容，多轮迭代直到任务完成。

## ⚠️ 核心约束：代码架构（高内聚低耦合）

> 业务逻辑必须拆分。单一脚本 / 模块不能过长。模块内部高内聚，模块之间低耦合。
> 新增于 2026-05-12。

### 规则总览

|| 规则 | 说明 |
|------|------|
| **单一职责** | 每个模块（`.py` 文件）只负责一件事 |
| **文件行数上限** | 单个 `.py` 文件 ≤ 500 行；超过必须拆分 |
| **禁止上帝模块** | 不得出现 `utils.py`、`helpers.py` 堆积无关联函数 |
| **层级调用禁止逆向** | 上层模块可调用下层，下层模块不得引用上层 |
| **依赖方向等于调用方向** | A 模块 import B 模块，则 B 不能循环 import A |

### 模块分层架构

```
trade/                          # 业务层（最上层，调用所有下层）
├── api.py                      # HTTP 路由（≤ 800 行），仅做请求解析 + 响应组装
├── helpers.py                  # 跨模块共享的薄胶水函数（解析配置、构建 prompt）
├── prompt.py                    # System prompt 模板（纯数据）
│
├── company.py                   # 公司实体（单一职责：company CRUD）
├── library.py                   # 文档库实体（单一职责：library CRUD）
├── customer.py                  # 客户实体（单一职责：customer CRUD）
├── chat_memory.py               # 对话记忆（单一职责：conversation CRUD）
│
└── database.py                  # 数据访问层（单一职责：DB 连接 + schema + 迁移）
```

**分层说明**：

| 层级 | 模块 | 职责 | 可调用 |
|------|------|------|--------|
| L1（入口） | `api.py` | HTTP 解析、响应、SSE 流 | L2 |
| L2（业务） | `company`/`library`/`customer`/`chat_memory` | 业务实体 CRUD | L3 |
| L3（数据） | `database.py` | DB 连接、schema、迁移 | 无（纯工具） |
| L4（数据） | `helpers.py` / `prompt.py` | 配置解析、prompt 模板 | L3（仅 helpers） |

### 拆分触发条件

当一个模块满足以下任一条件时，**必须立即拆分**：

```
1. 总行数 > 500 行
2. 职责种类 > 1 种（不含纯数据结构）
3. 存在与当前模块无关的 import 链
4. 测试时需要 mock 超过 3 个外部依赖
```

### 拆分操作规范

**问题模块** → **拆分步骤**：

```
1. 识别该模块承担的所有职责
2. 按职责拆分为独立 .py 文件（单一职责）
3. 新模块放置在同一层或下一层（依赖方向向下）
4. 原模块降级为薄转发层（仅 import + re-export）
5. 确保无循环依赖
```

**示例**：假设 `chat_memory.py` 膨胀到 600 行，同时处理 conversation CRUD 和 hindsight 调用：

```
chat_memory.py (600 行)
    ├── chat_memory.py          # conversation CRUD（降级为薄层，≤ 100 行）
    └── memory.py                # Hindsight 客户端封装（新增 L2 模块）
```

### 禁止模式

```
❌ utils.py（万能工具箱，积累无关联函数）
❌ helpers.py 堆积 > 200 行
❌ api.py 出现业务逻辑（DB 操作、实体判断）
❌ L3 模块（L2 的下层）import L2 模块
❌ 跨层调用链：api.py → database.py → api.py（循环）
```

### 代码注释规范

> 代码中必须包含充分的中文说明文字，使阅读者无需阅读实现细节即可理解意图和逻辑。
> 新增于 2026-05-12。

**所有函数必须有 docstring**：

```python
def get_library(library_id: int, company_id: int) -> dict | None:
    """根据 ID 获取指定公司的文档库。

    Args:
        library_id: 文档库主键 ID
        company_id: 公司主键 ID（用于数据隔离校验）

    Returns:
        文档库字典（含 id/name/root_path/description/created_at）；
        不存在或不属于该公司时返回 None。

    Raises:
        不会抛异常，所有错误情况均返回 None。
    """
```

**复杂逻辑必须分行注释**：

```python
# ❌ 差：无注释
result = [x for x in items if x.get("active") and x.get("company_id") == cid]

# ✅ 好：分行注释
active_items = [x for x in items if x.get("active")]      # Step 1: 筛选激活状态
company_items = [x for x in active_items                   # Step 2: 按公司隔离
                 if x.get("company_id") == cid]
```

**每个 if/分支必须有说明**：

```python
if not agent_identity:
    # 无公司身份配置时，使用通用系统提示词（降级策略）
    system_parts = [TRADE_SYSTEM_PROMPT]
else:
    # 注入公司身份：公司名称、产品范围、销售风格
    system_parts = [TRADE_SYSTEM_PROMPT, "", "## 公司身份配置", agent_identity]
```

**每章节代码块之间用注释分隔**：

```python
# =====================================================================
# 第一部分：健康检查（Provider 检查 → 模型配置检查）
# =====================================================================

# =====================================================================
# 第二部分：构建查询（公司身份 → prompt 组装 → 库上下文注入）
# =====================================================================
```

**注释覆盖率要求**：

|| 代码类型 | 最低要求 |
|---------|---------|
| 函数 docstring | 全部（含 private 函数） |
| if/分支 | 每个分支均有说明 |
| 循环 | 复杂循环有行内注释 |
| 异常处理 | 说明为何 try/except 及捕获后的处理策略 |
| API 路由 | 行末说明业务含义（如 `# 公司 ID 必填，否则无法隔离数据`） |
| 关键业务逻辑 | 注释说明决策依据和业务规则 |

**CI 检查**（可选，用 `radon` 或 ` lizard`）：

```bash
# 检查注释密度（comment ratio），低于 15% 报警
radon cc -a trade/ -e "migrations|__pycache__"
```

### 安装前预检（pre_install_check）

由于 `pip install trade` 不会自动安装 hermes-agent（依赖声明只是声明，不触发安装顺序），
Trade 提供 `pre_install_check.py`，在安装 trade **之前**主动检测 hermes-agent 状态：

```
安装 trade 之前
    │
    └─ python pre_install_check.py
           │
           ├─ hermes-agent 未安装 → exit 1 + 打印 chefroger fork 安装说明
           ├─ 已安装但来源错误（NousResearch） → exit 2 + 卸载重装说明
           ├─ 版本过低（< 0.12.0） → exit 2 + 升级说明
           └─ 版本兼容（>= 0.12.0，chefroger fork） → exit 0 → 继续安装 trade
```

**退出码**：

| 退出码 | 含义 |
|--------|------|
| 0 | 兼容，继续 `pip install trade` |
| 1 | hermes-agent 未安装，请先安装 |
| 2 | hermes-agent 版本不兼容或来源错误 |

**安装 chefroger fork 的 Hermes**：

```bash
# 官方一键安装（使用 chefroger fork 而非 NousResearch）
curl -fsSL \
  https://raw.githubusercontent.com/chefroger/hermes-agent/main/scripts/install.sh \
  | bash

# 或手动克隆
git clone --branch main \
  https://github.com/chefroger/hermes-agent.git \
  ~/.hermes/hermes-agent
cd ~/.hermes/hermes-agent
uv pip install -e .   # 或 pip install -e .
```

**预检脚本位置**：项目根目录 `pre_install_check.py`（343 行，stdlib-only，无外部依赖）。

## 三、多公司架构（Multi-Tenancy）

> 每个公司有独立的 AI 身份、文档库、客户和对话记录，数据完全隔离。
> 新增于 2026-05-11。

### 架构概述

```
companies ──────────── trade_companies
    │                        │
    │ 1:N                    │ 1:1 (每个公司对应一条 Trade 配置记录)
    ▼                        ▼
libraries ──────────── conversations
    │
    ▼
customers
```

**数据隔离**：所有 API 端点和数据库查询均通过 `company_id` 过滤，跨公司数据完全隔离。

### 数据库变更

|| 表 | 变更 | 说明 |
|----|------|------|
|| `companies` | 新增 | 公司基本信息（name, slug, contact, is_active） |
|| `trade_companies` | 新增 | Trade 专属配置（data_dir, agent_identity_md） |
|| `libraries` | +company_id | FK → companies(id)，级联删除 |
|| `customers` | +company_id | FK → companies(id)，级联删除 |
|| `conversations` | +company_id | FK → companies(id)，级联删除 |

**备用列（Spare Columns）**：所有表均预留 3 个 TEXT 类型备用列（`extra1`/`extra2`/`extra3`），默认 `'{}'`，用于存储 JSON 结构的扩展字段，避免未来 schema 变更需要 ALTER TABLE。所有 6 张表共计 18 个备用列。

|| 表 | extra1 | extra2 | extra3 |
|----|------|--------|--------|--------|
|| `companies` | `{"industry":"","country":""}` | `{"employee_count":"","annual_revenue":""}` | 预留给未来扩展 |
|| `trade_companies` | `{"max_iterations":90,"temperature":0.7}` | `{"model":"","provider":""}` | 预留给未来扩展 |
|| `libraries` | `{"scan_depth":3,"file_count":0,"last_scan":""}` | `{"indexed":false,"index_version":1}` | 预留给未来扩展 |
|| `customers` | `{"country":"","website":"","linkedin_url":""}` | `{"customer_type":"","tier":"","source":""}` | 预留给未来扩展 |
|| `customer_libraries` | `{"relevance_score":0.0,"notes":""}` | 预留给未来扩展 | 预留给未来扩展 |

## 三、核心业务功能

### 3.1 引流 — 找客户

#### 3.1.1 B2B 平台诊断与优化

用户提供阿里国际站 / 中国制造网 / 独立站的产品链接，Agent 诊断平台店铺并给出改进方案：
- 分析产品标题、关键词、描述的质量和覆盖度
- 检查产品图片、视频是否符合平台最佳实践
- 对比同行 Top 店铺，指出差距和优化方向
- 输出可直接执行的产品发布 / 修改清单

**输入**：平台店铺链接 或 产品链接
**输出**：诊断报告 + 改进方案（标题优化、关键词建议、描述模板、图片要求）

#### 3.1.2 社媒内容策略与素材生成

用户提供同行社媒账号链接（Facebook / Instagram / TikTok / YouTube），Agent 学习同行内容策略后，结合本公司产品生成社媒发布方案：
- 分析同行的内容类型、发布频率、互动数据、标签策略
- 生成适用于本公司产品的社媒内容日历（周 / 月计划）
- 提供可直接发布的具体帖文（文案 + 图片建议 + 标签）

**输入**：同行社媒链接（可多个）+ 本公司产品资料
**输出**：社媒发布方案 + 内容日历 + 具体帖文（可直接复制粘贴发布）

#### 3.1.3 领英营销方案

用户提供本公司产品资料，Agent 结合领英 B2B 开发方法，生成完整的领英营销方案：
- **客户画像**：基于产品特征定义目标客户（职位、行业、地区、公司规模）
- **个人账号设置**：头像、背景图、标题、About 部分的内容模板
- **公司主页设置**：公司介绍、产品展示、员工邀请策略
- **年度营销方案**：包含计划时间表和具体执行内容，可直接复制粘贴使用
- **内容素材库**：行业洞察帖、产品价值帖、客户案例帖的模板库

**输入**：本公司产品册 / 参数表 + 行业资料
**输出**：客户画像 + 账号设置模板 + 公司主页内容 + 年度营销方案（含计划表）

#### 3.1.4 海关数据 / 广交会数据挖掘

用户上传海关数据（进出口记录）或广交会参展商数据，Agent 自动筛选：
- 从数据中提取本行业的目标客户（采购商 / 进口商）
- 分析客户采购规律（频次、数量、品类、季节）
- 按优先级排序输出目标客户列表
- 给出每个客户的开发建议（切入点、推荐产品、预期订单量）

**输入**：海关数据文件（CSV / Excel）+ 本公司产品范围
**输出**：本行业目标客户列表 + 开发优先级 + 每个客户的开发策略

#### 3.2.1 AI 高级销售员

#### 3.2.2 客户分析与开发信

- 首次沟通的 talking points 和常见问题预案

**输入**：领英客户链接 或 客户信息文本（公司名 + 联系人 + 职位）
**输出**：客户分析报告 + 开发信 / 消息 + 跟进计划

### 3.5 自动化定时任务（业务员每日工作引擎）

> 详细设计见 [外贸业务知识库.md - 第十二章](外贸业务知识库.md)

将外贸业务员的每日工作规律固化为 Hermes cron 定时任务，Agent 按时触发、自动准备内容、业务员只需执行和复制粘贴。

**核心机制**：利用 Hermes 内置 cron 系统（`cron.create_job`），根据外贸业务员的每日工作流程，自动创建 7 个定时任务。Agent 到时间自动执行 prompt，生成可执行的内容推送给业务员。

**7 个定时任务一览**：

| 时间 | 任务 | 触发动作 |
|------|------|---------|
| 08:00（工作日） | 早安简报 | Agent 总结昨日互动 + 今日待跟进清单 + 客户时区窗口提醒 |
| 09:30（工作日） | 领英内容发布 | Agent 生成本日 Post + Article 草稿（可直接复制粘贴发布） |
| 10:00（工作日） | 欧洲客户开发信 | Agent 筛选欧洲客户 → 生成跟进邮件 → 业务员发送 |
| 11:00（工作日） | B2B 平台数据检查 | Agent 检查平台询盘、产品排名、同行动态 |
| 14:00（工作日） | 社媒内容发布 | Agent 生成 FB / Ins / TikTok 帖子 |
| 16:00（工作日） | 北美客户开发信 | Agent 筛选北美客户 → 生成跟进邮件 → 业务员发送 |
| 17:30（工作日） | 每日工作总结 | Agent 汇总今日数据 → 追加到工作日报 |
| 17:30（每周五） | 周报生成 | Agent 汇总本周日报 → 生成周报 |

**初始化方式**：
```
trade cron setup-trade-tasks
```
一键创建全部定时任务，无需手动配置。

### 3.11 OSINT 客户背景调查与风险评估（升级版背调）

> 新增于 2026-05-12。源自客户背调对话中的真实需求：单一邮箱平台检测不够，需要多信号交叉验证。

#### 3.11.1 现状与差距

**当前 `b2b-email-intel` 的能力**：
- 仅检测邮箱在 121+ 主流平台的注册状态（`holehe`）
- 输出：平台名 | 是否注册 | 用户名 | 档案 URL

**对话揭示的真实背调需求**——完整尽职调查需要 5 个维度交叉验证：

| 验证维度 | 当前 | 目标 |
|---------|------|------|
| 邮箱平台注册 | ✅ `holehe` | 保留 |
| 域名注册信息 | ❌ 缺失 | WHOIS 查询（注册时间/注册商/到期时间） |
| 企业邮箱验证 | ❌ 缺失 | `@公司.com` vs Gmail/QQ 个人邮箱判断 |
| 网站技术栈 | ❌ 缺失 | BuiltWith 检测（WordPress/Shopify/自建） |
| 制裁名单筛查 | ❌ 缺失 | OFAC/UN/欧盟/中国制裁名单交叉比对 |

**核心洞察（来自对话）**：单一信号不够，需要多信号交叉验证。例如：
- LinkedIn 主页看起来真，但域名 3 个月前新注册 → 红旗
- 企业邮箱用的是 `gmail.com` 而非 `@公司.com` → 红旗
- 网站技术栈是免费博客平台 → 红旗

#### 3.11.2 多层验证架构

```
用户输入：公司名 / 邮箱 / 网站 URL
    │
    ├─────────────────────────────────────────────────────────────┐
    │  Layer 1: 邮箱基础检测（holehe，已有）                         │
    │  ├─ holehe 邮箱注册检测（121+ 平台）                          │
    │  └─ 输出：found_count, 社交档案 URL 列表                      │
    └─────────────────────────────────────────────────────────────┘
                          ↓
    ├─────────────────────────────────────────────────────────────┐
    │  Layer 2: 域名深度查询（新增）                                 │
    │  ├─ WHOIS 查询：注册时间 / 注册商 / 到期时间 / DNS 服务器      │
    │  ├─ 注册时间 < 1 年 → ⚠️ 新注册域名（骗子特征）                 │
    │  ├─ 注册时间 > 3 年 → ✅ 老域名（正常）                        │
    │  └─ BuiltWith 技术栈检测                                      │
    └─────────────────────────────────────────────────────────────┘
                          ↓
    ├─────────────────────────────────────────────────────────────┐
    │  Layer 3: 企业邮箱验证（新增）                                 │
    │  ├─ 提取邮箱域名：xxx@company.com → company.com               │
    │  ├─ 对比网站域名：company.com vs company.com                  │
    │  ├─ @gmail.com / @qq.com / @163.com → 🚩 个人邮箱红旗         │
    │  └─ @企业域名 → ✅ 企业邮箱（加分项）                          │
    └─────────────────────────────────────────────────────────────┘
```

#### 3.11.4 风险评分体系

**交叉验证评分卡**（每项独立打分，汇总为综合评级）：

```
✅ 加分项                              ⚠️ 减分项
─────────────────────────────────────────────────────
域名 > 3 年                           域名 < 1 年
有企业邮箱 (@公司.com)                  只用 gmail/qq/163
LinkedIn 公司主页存在且员工多           无 LinkedIn 公司页
网站技术栈正规（企业级）                 免费博客/临时建站工具
域名年龄 ≈ 公司成立时间                  域名比公司还老很多
WHOIS 注册信息完整                     WHOIS 隐私保护/信息缺失
不在任何制裁名单                        在任一制裁名单中出现
```

**综合评级定义**：

| 评级 | 条件 | 建议 |
|------|------|------|
| 🟢 低风险 | 加分项 ≥ 4 且无红旗 | 正常跟进 |
| 🟡 中风险 | 有 1-2 个红旗 | 谨慎跟进，要求更多验证 |
| 🔴 高风险 | 红旗 ≥ 3 或任一制裁名单命中 | 强烈建议拒绝，或高层特批 |

## 五、输出质量标准

### 5.1 语言策略

- **单文档单语言**：面向海外客户全英文，面向国内客户全中文。禁止中英文混排
- 产品型号、SKU 保留原文不翻译

### 5.2 内容标准

- 所有输出内容（帖文、开发信、营销方案）**可直接复制粘贴使用**，不需要用户再修改格式
- 引用来源时标注 `📄 {文件名} | Sheet: {sheet} | Row: {行范围}`
- 不确定的信息明确告知用户，不编造数据
- 无占位符（"XXX"、"Lorem ipsum"、"[TBD]"），空缺信息宁可省略

### 5.3 业务文档生成规范（PPTX / DOCX / XLSX）

| 维度 | 标准 |
|------|------|
| 配色 | 最多 3 色，与行业匹配。不用通用蓝色 |
| 字号 | 标题 36-44pt，段落标题 18-24pt，正文 12-16pt |
| 表格 | 完整数据填充，深色表头 + 交替行背景 |
| 布局 | 每页有视觉元素（色条、卡片、图标），禁止纯文本框 |

### 5.4 行业知识

- 贸易术语：FOB / CIF / EXW / MOQ / Lead Time / T/T / L/C / Incoterms
- 货币识别：¥/CNY、$/USD、€/EUR、£/GBP
- 单位换算：mm ↔ inch、kg ↔ lb、MPa ↔ psi、°C ↔ °F
- 平台知识：阿里国际站运营规则、领英 B2B 开发方法、社媒内容最佳实践

## 3.12 Prompt 隔离架构（用户可编辑的 System/Skill Prompt）

> 新增于 2026-05-12。系统所有注入 LLM 的 prompt 均支持用户编辑，文件优先、DB 缓存。

### 3.12.1 架构原则

| 原则 | 说明 |
|------|------|
| **文件是 Source of Truth** | `~/.trade/` 下的 `.md` 文件是 prompt 的权威来源 |
| **DB 是运行时缓存** | `agent_identity_md` 等字段为文件的热缓存，文件变更自动生效 |
| **mtime 缓存** | 避免每次请求重复读文件，文件变更后自动失效重载 |
| **Skill 从 SKILL.md 读** | `injection_prompt` 存在于 `SKILL.md` frontmatter，用户改 skill 文件即改行为 |

### 3.12.2 Prompt 文件布局

```
~/.trade/                                  # 运行时数据（.gitignore）
├── prompts/
│   └── system.md                         # 全局 system prompt（用户可编辑）
├── companies/{slug}/
│   └── agent_identity.md                 # 公司级 agent identity（覆盖全局 system）
└── companies.db                          # agent_identity_md 字段 = 热缓存

~/.hermes/skills/b2b-osint/
└── SKILL.md                              # injection_prompt 在 frontmatter
```

### 3.12.3 读取优先级

**System Prompt（纵向覆盖，取最高优先级）：**
1. `~/.trade/companies/{slug}/agent_identity.md` — 公司级
2. `~/.trade/prompts/system.md` — 全局自定义
3. 代码 fallback（`trade/prompt.py` 的 `TRADE_SYSTEM_PROMPT`）

**Skill Injection（纵向覆盖，取最高优先级）：**
1. `~/.hermes/skills/{skill}/SKILL.md` frontmatter 的 `injection_prompt`
2. 代码 fallback（`skill_router.py` 的 `_SKILLS` 字典）

**最终拼接（横向）：**
```
system_prompt + skill_injection + doc_context + user_query
```

### 3.12.4 文件变更自动生效

`trade/prompts.py` 维护 mtime 缓存（`_PROMPT_CACHE`）：
- 文件 `mtime` 未变 → 直接用缓存
- 文件 `mtime` 变化 → 重新读文件，更新缓存
- 用户 `vim ~/.trade/prompts/system.md` 后下次请求自动生效，无需重启

### 3.12.5 实施文件

| 文件 | 作用 |
|------|------|
| `trade/prompts.py` | 统一 prompt 文件加载器（mtime 缓存） |
| `trade/skill_router.py` | 从 SKILL.md frontmatter 读取 `injection_prompt` |
| `trade/helpers.py` | 改用 `prompts.resolve_system_prompt()` |
| `trade/onboarding.py` | 写 identity 文件 + DB 双写 |
| `.trade-template/prompts/system.md` | system prompt 安装模板 |
| `.trade-template/companies/company-slug/agent_identity.md` | agent identity 安装模板 |


## 3.13 对话长期记忆与历史查询

> 新增于 2026-05-12。解决"今天不知昨天事"——所有对话永久带时间戳存储，Agent 可随时主动查询。

### 3.13.1 核心原则

- **所有对话永久存**：不压缩、不摘要、不丢弃，`created_at` 是必填字段
- **默认注入少量最近历史**：10-20 轮（~10k tokens），保证近期上下文连贯
- **更早历史按需查询**：Agent 通过工具主动查，LLM 根据工具描述自行判断时机
- **阈值由代码控制**：用户完全不可感知
- **Hermes 原生记忆互通**：`save_with_context()` 每次保存对话后，自动将摘要写入 `~/.hermes/memories/MEMORY.md`，Hermes Agent 跨项目也能记住 B2B 业务对话

### 3.13.2 Token 预算分配

| 区段 | 上限 | 说明 |
|------|------|------|
| System prompt | ~5k | 公司 identity + doc context |
| Skill injection | ~1k | 当前触发的 skill augment |
| **Recent history** | **~10k** | **固定 10-20 轮，最近上下文** |
| Current query | ~1k | 用户本次问题 |
| Reserved space | ~83k | 留给 Agent 推理和响应 |

### 3.13.3 Prompt 构建逻辑（build_query）

**阈值判断对象：整个 prompt 的 token 总数**

```python
# 阈值硬编码，不暴露给用户
if total_tokens < 20_000:
    history_limit = 20   # 充足空间，注入更多历史
    hint = None
elif total_tokens < 50_000:
    history_limit = 10    # 开始压缩历史量
    hint = "如需更早的对话历史，请使用 chat_memory_list 工具。"
else:
    history_limit = 5     # 强制精简
    hint = "当前上下文较长，历史对话已精简。如需查询更早内容，请使用 chat_memory_list 工具。"
```

**History 注入始终为固定轮数（history_limit），不随阈值无限增长。**

### 3.13.4 工具描述驱动（无触发词）

Agent 何时查历史不由触发词决定，而由工具描述决定：

```json
{
  "name": "chat_memory_list",
  "description": "查询与当前公司的历史对话记录。当用户提到'之前''上次''以前'等时间相关词汇，
                 或询问过去讨论过的内容时，使用此工具检索相关历史。",
  "parameters": {
    "time_range": {
      "type": "string",
      "enum": ["today", "this_week", "this_month", "all"],
      "description": "时间范围：今天/本周/本月/全部"
    },
    "limit": {
      "type": "integer",
      "default": 20,
      "description": "返回的最大消息数"
    }
  }
}
```

LLM 读取 SKILL.md 的工具描述后，自行判断何时需要调用——用户说"你上次推荐的那个供应商叫什么"时，LLM 会主动查 `chat_memory_list`。

### 3.13.5 Hindsight 与 SQLite 的分工

| 查询类型 | 示例 | 存储/检索 |
|---------|------|---------|
| **时序查询** | "昨天的对话"、"上周谈了什么" | SQLite `ORDER BY created_at` |
| **语义查询** | "那个做芯片的供应商"、"之前提到的方案" | Hindsight 向量检索 |

- 有 Hindsight API Key → 时序用 SQLite，语义用 Hindsight
- 无 Hindsight API Key → 语义查询降级为 SQLite `WHERE query LIKE '%关键词%'`

## 十二、升级逻辑（数据完整性保护）

> Trade 升级时必须保证原有业务数据（数据库、公司目录文件）完整可用，
> 不能因 schema 变更而丢失任何用户数据。新增于 2026-05-12。

### 12.1 设计原则

| 原则 | 说明 |
|------|------|
| **向后兼容** | 新版本代码必须能读写旧版本写入的数据 |
| **自动迁移** | 启动时自动检测并执行迁移，无需用户手动干预 |
| **原子性** | 迁移失败时数据库回滚，不留下半成品状态 |
| **幂等性** | 迁移脚本可安全重复执行，不会重复插入数据 |
| **可逆性** | 迁移脚本有清晰的 `up()` / `down()` 逻辑（用 Python 而非 SQL 注释实现） |

### 12.2 迁移检测机制

每次 `init_db()` 执行时，数据库层按以下顺序检测当前 schema 版本：

```
1. 读取 PRAGMA table_info(libraries) — 获取当前列集合
2. 判断 "company_id" 列是否存在
   ├─ 不存在 → v0 schema（单公司版本）
   └─ 存在   → v1 schema（多公司版本）
3. 后续版本检测同理，逐版本递进
```

**重要**：不使用独立的 `schema_version` 表，而是通过检测关键列是否存在来判断版本。
原因：避免多表写入增加原子性风险；列存在即代表该版本迁移已完成。

### 12.3 版本演进与迁移路径

#### v0 → v1：单公司 → 多公司（Multi-Tenancy）

**触发条件**：检测到 `libraries` 表无 `company_id` 列。

**迁移前数据状态**（v0）：
```
libraries:     id, name, root_path, description, created_at, updated_at
customers:     id, name, contact, note, created_at, updated_at
conversations: id, query, response, files_read, created_at
```

**迁移后数据状态**（v1）：
```
companies:        id, name, slug, logo_url, website, contact_name, ...,
                 extra1='{}', extra2='{}', extra3='{}'
trade_companies: company_id(PK), data_dir, agent_identity_md, is_active, created_at,
                 extra1='{}', extra2='{}', extra3='{}'
libraries:       + company_id (FK → companies(id), DEFAULT 1),
                 extra1='{}', extra2='{}', extra3='{}'
customers:       + company_id (FK → companies(id), DEFAULT 1),
                 extra1='{}', extra2='{}', extra3='{}'
conversations:   + company_id (FK → companies(id), DEFAULT 1),
                 extra1='{}', extra2='{}', extra3='{}'
```

**迁移操作步骤**（`database.py:_migrate_from_v0()`）：

```
Step 1: 创建新表
  CREATE TABLE companies (...)
  CREATE TABLE trade_companies (...)
  INSERT OR IGNORE INTO companies (name='我的公司', slug='default', ...)
  INSERT OR IGNORE INTO trade_companies (company_id=1, data_dir='', ...)

Step 2: ALTER TABLE（逐表添加列）
  ALTER TABLE libraries     ADD COLUMN company_id INTEGER REFERENCES companies(id) ON DELETE CASCADE
  ALTER TABLE customers     ADD COLUMN company_id INTEGER REFERENCES companies(id) ON DELETE CASCADE
  ALTER TABLE conversations ADD COLUMN company_id INTEGER REFERENCES companies(id) ON DELETE CASCADE

Step 3: 填充 company_id（现有数据全部归属公司 1）
  UPDATE libraries    SET company_id = 1
  UPDATE customers    SET company_id = 1
  UPDATE conversations SET company_id = 1

Step 4: 创建索引
  CREATE INDEX idx_libraries_company     ON libraries(company_id)
  CREATE INDEX idx_customers_company     ON customers(company_id)
  CREATE INDEX idx_conversations_company ON conversations(company_id)

Step 5: 补充备用列（所有 6 张表）
  _add_spare_columns(conn)  # 幂等 ADD COLUMN，已存在则静默忽略
```

**数据完整性保证**：
- 所有现有数据自动归属公司 1（"我的公司"），不丢失任何记录
- 新表与旧表数据在同一个事务中完成，失败则整体回滚
- `ON DELETE CASCADE` 确保删除公司时自动清理其全部业务数据

---

#### v1 → v2（示例，未来扩展）

未来每个 schema 变更都遵循相同模式，在 `database.py` 中追加新的 `_migrate_v1_to_v2()` 函数：

```python
def _migrate_v1_to_v2(conn):
    # Step 1: 检测是否需要迁移（幂等性）
    cur = conn.execute("PRAGMA table_info(libraries)")
    cols = {row[1] for row in cur.fetchall()}
    if "new_column" in cols:
        return False  # 已迁移

    # Step 2: 执行迁移（在同一个事务中）
    conn.execute("ALTER TABLE libraries ADD COLUMN new_column TEXT DEFAULT ''")
    # ... 其他操作

    # Step 3: 补充备用列（未来每张新表均需）
    _add_spare_columns(conn)

    conn.commit()
    return True
```

### 12.4 目录文件迁移

公司数据目录（`~/.trade/{slug}/`）的迁移在 `company.py:_ensure_data_dir()` 中处理：

```
触发时机：company.create() 或 company.get() 时调用
触发条件：目标目录 ~/.trade/{slug}/ 不存在

迁移操作：
  1. 读取 .trade-template/ 作为源
  2. 在临时目录中复制模板
  3. 重命名占位符目录：
       companies/company-slug  → companies/{slug}
       libraries/library-slug  → libraries/{slug}
       clients/client-slug     → clients/{slug}
  4. 移动临时目录 → ~/.trade/{slug}/
```

**数据完整性保证**：
- `if target.exists(): return target` — 已存在的公司目录永远不会被覆盖
- 新公司使用模板创建，不影响已有公司的目录文件
- `shutil.move()` 是原子操作（在同一文件系统内）

### 12.5 升级失败处理

| 失败场景 | 处理方式 |
|---------|---------|
| 数据库迁移 SQL 执行失败 | `conn.rollback()` 回滚，进程退出并输出错误信息 |
| 公司目录创建失败 | `company.create()` 抛出异常，DB 事务未提交 |
| 模板目录不存在 | `_ensure_data_dir()` 静默创建空目录，不阻止公司创建 |
| 磁盘空间不足 | Python IOError，进程退出，DB 未写入 |

**日志输出**：
- 迁移开始：`Detected v0 schema — migrating to multi-company v1 …`
- 迁移完成：`Migration complete. Default company '我的公司' (slug=default) created.`
- 迁移失败：完整的 SQLite 错误信息，便于诊断

### 12.6 版本降级（Downgrade）

Trade **不支持**从高版本降级到低版本。降级操作需要用户手动：
1. 备份 `~/.trade/` 目录和 `data/trade.db`
2. 重新安装旧版本 pip 包
3. 手动还原数据

降级保护原因：ALTER TABLE 不可逆（SQLite 不支持 DROP COLUMN），降级可能导致数据丢失。

### 12.7 升级前检查清单（维护者用）

每次发布新 schema 变更前，必须确认：

```
[ ] 新迁移函数已实现 up() 逻辑，幂等性已验证
[ ] 在旧版本数据库上执行迁移，所有现有数据不丢失
[ ] 新版本数据库回滚到迁移前状态，验证 rollback 有效
[ ] 迁移后公司目录结构正确（companies/{slug}/ 等目录名已重命名）
[ ] 前端能正确访问迁移后的公司数据（X-Company-ID 路由正常）
[ ] 单元测试覆盖迁移路径（test_v0_to_v1_migration）
```
